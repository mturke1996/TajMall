# تنبيه أمني عاجل — يتطلب إجراءً منك الآن

تم اكتشاف أن ملف `.env.example` كان يحتوي (منذ أول commit في هذا المستودع)
على قيم **حقيقية** بدلاً من قيم توضيحية: مفتاح `service_role`، كلمة مرور
قاعدة البيانات، و`KEEPALIVE_SECRET`. المستودع
[github.com/mturke1996/TajMall](https://github.com/mturke1996/TajMall) عام،
بمعنى أن هذه القيم كانت مكشوفة لأي شخص على الإنترنت.

تم تنظيف `.env.example` في هذا الإصلاح (استبدال القيم الحقيقية بقيم
توضيحية)، **لكن هذا وحده لا يكفي** — القيم القديمة موجودة في تاريخ Git
للأبد ولا يمكن اعتبارها آمنة بعد الآن. **الحل الوحيد الحقيقي هو تدوير
(تجديد) كل القيم التالية من لوحة تحكم Supabase/Vercel مباشرة.**

## ما يجب عليك فعله الآن (بالترتيب)

### 1. تدوير مفتاح service_role و anon/publishable

1. Supabase Dashboard → مشروعك → **Settings → API**
2. اضغط **"Roll"** أو **"Reset"** بجانب `service_role` (أو `secret keys` في
   الواجهة الجديدة). سيتوقف المفتاح القديم عن العمل فوراً.
3. افعل الشيء نفسه لمفتاح `anon` / `publishable` إن كان متاحاً.
4. انسخ القيم الجديدة.

### 2. تغيير كلمة مرور قاعدة البيانات

1. Supabase Dashboard → **Settings → Database → Database password**
2. اضغط **"Generate a new password"** واحفظ القيمة الجديدة فوراً.
3. حدّث `DATABASE_URL` و`DIRECT_URL` بكلمة المرور الجديدة (رمّز الأحرف
   الخاصة بترميز URL إن وُجدت — مثلاً `%40` بدل `@`).

### 3. توليد KEEPALIVE_SECRET جديد

قيمة عشوائية طويلة، مثلاً بتنفيذ `openssl rand -hex 32` محلياً، أو أي
مولّد كلمات مرور عشوائية موثوق.

### 4. تحديث كل مكان يحتفظ بالقيم القديمة

- ملف `.env.local` المحلي (غير مرفوع لـ git، لكنه يحتوي القيم القديمة الآن).
- متغيرات البيئة على **Vercel** → Project Settings → Environment Variables
  (لكل من `Production` و`Preview` و`Development`).
- **GitHub Actions** → Settings → Secrets and variables → Actions → السرّ
  `KEEPALIVE_SECRET`.
- أعد نشر (Redeploy) المشروع على Vercel بعد تحديث المتغيرات.

### 5. (اختياري لكن مستحسن) راجع سجلات الاستخدام

من Supabase Dashboard → **Reports / Logs** تحقق من عدم وجود استعلامات أو
اتصالات غريبة حدثت بين تاريخ أول commit والآن، تحسباً لاستخدام غير مشروع
للمفتاح القديم.

## ثغرة إضافية أُصلحت في الكود (لا تحتاج إجراءً منك)

بالإضافة إلى تسريب المفاتيح، كانت توجد ثغرة ترقية صلاحيات (privilege
escalation): أي مستخدم مسجّل (حتى بدرجة "viewer") كان يستطيع تغيير دوره
الخاص إلى `owner` مباشرة من المتصفح، ومسار `/api/admin/invite` كان يقبل
طلبات من أي مستخدم مسجّل بدون فحص صلاحية. تم إصلاح كلتا الثغرتين:

- **قاعدة البيانات**: `supabase/migrations/040_prevent_role_self_escalation.sql`
  — يجب تطبيقه في **Supabase → SQL Editor** (نفس طريقة كل الهجرات السابقة)
  قبل أن يصبح الإصلاح فعّالاً.
- **الكود**: `src/app/api/admin/invite/route.ts` — يفرض الآن أن يكون
  المستدعي owner/admin، وأن owner فقط يستطيع منح دور owner لغيره.

بعد تطبيق الهجرة 040، ستنتفي المشكلة نهائياً حتى إن حاول أحد استدعاء
واجهة Supabase مباشرة (لا تعتمد على واجهة التطبيق فقط).
