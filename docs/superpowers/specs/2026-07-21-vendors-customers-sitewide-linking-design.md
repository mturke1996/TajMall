# ربط الموردين والعملاء عبر كامل الموقع والخدمات — مواصفة

**الحالة:** معتمد للتنفيذ  
**التاريخ:** 2026-07-21  
**النطاق المعتمد من الحوار:** المسار **ب + ج** معاً

## الهدف

جعل `VENDOR` و `CUSTOMER` أطرافاً من الدرجة الأولى عبر كامل Fluxen: دخول موحّد، ربط بالمعاملات والسندات واليومية، تواصل هاتف/واتساب، تقارير إنفاق/إيراد، وربط إيرادات الخدمات (مواقف/إعلانات/خدمات) بعملاء بدون عقد إيجار، ومصروفات الخدمات بموردين.

## غير ضمن النطاق

- نظام مكالمات/VoIP/مركز اتصال (غير موجود في المنتج)
- فصل جداول Prisma القديمة `Customer`/`Vendor` عن `contacts` (تبقى مهملة)
- ذمم مدينة كاملة بنظام فواتير منفصل للعملاء غير المستأجرين (نكتفي بتجميع المعاملات المرتبطة)

## المعمارية

المصدر الوحيد للحقيقة: `public.contacts` مع `kind ∈ {VENDOR, CUSTOMER, TENANT, EMPLOYEE, OTHER}`.

```
contacts (VENDOR | CUSTOMER)
  ├── transactions.contact_id
  ├── journal_lines.contact_id
  ├── disbursement_vouchers.contact_id  (جديد، اختياري)
  └── تقارير التجميع حسب contact_id + kind + فئة الحساب
```

لا نُنشئ جداول أطراف جديدة. نُوسّع المسارات والواجهات والهجرات حول `contacts`.

## القرارات

1. **مسارات الدخول:** `/customers` يطابق `/vendors` — تحويل إلى `peopleSegmentHref('CUSTOMER')`. إضافة `/customers` إلى `LEGACY_PEOPLE_SEGMENT`. عنصر قائمة «العملاء» بجانب «الموردين».
2. **المعاملات:** شرائح فلتر `VENDOR` و `CUSTOMER` ظاهرة دائماً. عند نوع إيراد غير إيجار → افتراضي فلتر عملاء؛ عند مصروف → افتراضي موردين. إنشاء سريع يحترم الفلتر الحالي مع نص toast صحيح («مورد» / «عميل»).
3. **السندات:** عمود اختياري `disbursement_vouchers.contact_id → contacts(id)`. الإبقاء على `payee` كنص للعرض/PDF؛ عند اختيار جهة يُملأ `payee` من الاسم تلقائياً (قابل للتعديل).
4. **الهاتف/واتساب:** على صفحة جهة الاتصال والدليل — روابط `tel:` وواتساب لكل الأنواع (إعادة استخدام `src/lib/whatsapp.ts`)، وليس للمستأجرين فقط.
5. **خدمات بدون عقد (عملاء):** معاملات إيراد بفئات `REV-PRK` / `REV-SVC` / `REV-ADV` / `REV-KSK` / `REV-EVT` يمكن ربطها بـ `CUSTOMER` دون واجهة تخصيص إيجار. لا تُنشأ `tenant_charges` لهذه الحالة.
6. **خدمات بمورد (مصروف):** عند فئات تشغيل شائعة (تنظيف/أمن/صيانة إن وُجدت) يُقترح فلتر `VENDOR` ويُشجَّع اختيار مورد؛ لا نجعل الحقل إلزامياً في المرحلة الأولى لتجنب كسر التدفقات الحالية.
7. **التقارير:**
   - `/reports/vendor-spend` — إنفاق حسب المورد (YTD / فترة من فلتر التقارير الموحّد)
   - `/reports/customer-revenue` — إيراد حسب العميل (نفس الفترة)
   - بطاقات في مركز `/reports` + عناصر تنقل
8. **تفاصيل الجهة:** ملخص الفترة: مجموع معاملات مرتبطة + روابط مفلترة للمعاملات/اليومية + أزرار اتصال.

## معايير القبول

1. `/customers` و `/vendors` يفتحان دليل الناس بالشريحة الصحيحة مع إمكانية إضافة فورية.
2. يمكن إنشاء عميل ومورد من القائمة والدليل ورؤيتهما في التفاصيل.
3. معاملة مصروف يمكن ربطها بمورد؛ إيراد خدمة/مواقف يمكن ربطه بعميل دون عقد.
4. إذن صرف يمكن ربطه بمورد؛ PDF ما زال يعرض اسم المستفيد.
5. التقارير تعرض تجميعاً صحيحاً للفترة المختارة.
6. أزرار اتصال تعمل عند وجود رقم هاتف صالح.
7. لا انحدار على تدفقات المستأجرين/الإيجار الحالية.

## خطة التحقق

- اختبار يدوي على `npm run dev`: إنشاء جهات → معاملات → سند → تقارير
- اختبار وحدة/تكامل خفيف لمساعدات التجميع إن أُضيفت دوال نقية
- تطبيق migration السندات على بيئة التطوير قبل الاعتماد على الحقل الجديد
